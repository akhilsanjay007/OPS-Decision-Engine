# Ops Decision Engine

An end-to-end workspace for turning operational data into **actionable decisions** (triage, prioritization, routing, forecasting, and capacity planning).

## Dataset

This project uses the Kaggle dataset:

- [Multilingual Customer Support Tickets](https://www.kaggle.com/datasets/tobiasbueck/multilingual-customer-support-tickets)

### Expected file location

The notebooks expect the source CSV under `data/raw/`.

If you're using the Kaggle multilingual dataset, the CSV file is commonly named:

- `data/raw/aa_dataset-tickets-multi-lang-5-2-50-version.csv`

If you keep a different filename, update the notebook path accordingly.

## Data Workflow (Recommended)

Use a strict flow to keep outputs reproducible:

- **Raw → interim → processed**
- Put original files in `data/raw/` and do not edit them in place.
- Write deterministic cleaning steps that output to `data/interim/`.
- Write feature engineering / final dataset creation to `data/processed/`.

## Notebook Conventions (Windows)

- Run notebooks from the repo root (the folder that contains `data/`) to keep relative paths predictable.
- Prefer `pathlib` for filesystem paths.
- When writing outputs, ensure destination directories exist (e.g. `data/interim/`, `data/processed/`).

## RAG (retrieval)

The project includes a **similar-incident** retrieval stack: embeddings with [Sentence Transformers](https://www.sbert.net/) (`sentence-transformers/all-MiniLM-L6-v2` by default) and a persistent [Chroma](https://www.trychroma.com/) store under `artifacts/rag/chroma_db`, collection `incident_memory`.

### Data and index

1. Prepare the knowledge base (from the notebook above or your own pipeline): **`data/processed/rag_knowledge_base.jsonl`** — one JSON object per line with fields such as `doc_id`, `retrieval_text`, `issue_description`, `resolution`, `type`, `queue`, `priority`, `tags`, and `source`.
2. Build or refresh the vector index from the repo root:

   ```powershell
   python -m src.rag.build_index
   ```

   This embeds all `retrieval_text` values and writes the Chroma database to `artifacts/rag/chroma_db`.

### Query similar incidents (CLI)

```powershell
python -m src.rag.retrieve --query "Your issue description here" --top_k 5
```

Optional filters: `--queue "QUEUE NAME"`, `--type "TYPE"`. Override paths with `--chroma_path`, `--collection_name`, or `--model_name` if needed.

### Programmatic use

`src.rag.retrieve.retrieve_similar_incidents(...)` returns a list of dicts with `rank`, `doc_id`, `distance`, metadata fields (`issue_description`, `resolution`, `type`, `queue`, `priority`, `tags`), and `retrieval_text`. See `tests/test_retrieval.py` for examples.

## Decision pipeline

Two modules combine the **priority model** (Stage 5 SVM sklearn pipeline) with **RAG** and optionally an **LLM** for a written decision.

**Artifacts**

- **ML:** `artifacts/ml/priority_stage5_svm_pipeline.joblib` — produced by `python -m src.ml.train_stage5_svm` (evaluation artifacts also go under `outputs/ml/stage5_svm/`).
- **RAG:** Chroma DB at `artifacts/rag/chroma_db` (see [RAG (retrieval)](#rag-retrieval)).

### Predict priority + retrieve similar incidents

`src.pipeline.predict_and_retrieve.run_pipeline(...)` loads the joblib model, predicts priority from `issue_description`, `type`, and `queue`, enriches the retrieval query with lightweight signal extraction, and returns `predicted_priority`, `retrieved_incidents`, and `retrieval_query`.

CLI (from repo root):

```powershell
python -m src.pipeline.predict_and_retrieve --issue "Your text" --type "INCIDENT" --queue "CUSTOMER SERVICE" --top_k 3
```

Programmatic use and sample cases: `tests/test_predict_and_retrieve.py`.

### Full pipeline: predict + retrieve + LLM decision

`src.pipeline.predict_retrieve_generate.run_full_pipeline(...)` runs predict-and-retrieve, reranks incidents (queue/type/priority boosts), removes near-duplicates, computes a confidence score/level, and then calls the OpenAI Chat Completions API (`gpt-4o-mini` by default in code) to produce a structured operations decision.

**Setup:** set your API key in the environment before running:

```powershell
$env:OPENAI_API_KEY = "your-key-here"
```

CLI:

```powershell
python -m src.pipeline.predict_retrieve_generate --issue "Your text" --type "INCIDENT" --queue "CUSTOMER SERVICE" --top_k 3
```

Programmatic smoke tests: `tests/test_predict_retrieve_generate.py`.

### Pipeline evaluation

`tests/evaluate_pipeline.py` runs a fixed set of cases end-to-end and records:

- ML priority prediction
- rule-based recommended priority from retrieved evidence
- LLM priority extracted from generated output
- confidence score/level
- escalation recommendation and same-queue evidence ratio

Run:

```powershell
python -m tests.evaluate_pipeline
```

Output:

- `artifacts/evaluation/pipeline_evaluation_results.csv`
